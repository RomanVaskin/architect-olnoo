Детали отдельных слоёв постепенно выносятся в специализированные документы: [03-DATABASE.md](03-DATABASE.md), [04-AI-PIPELINE.md](04-AI-PIPELINE.md), [05-FRONTEND.md](05-FRONTEND.md), [06-BACKEND.md](06-BACKEND.md), [07-API.md](07-API.md) и [08-SECURITY.md](08-SECURITY.md).

## Architecture Principles

### AI-first

Платформа проектируется вокруг AI как основного механизма создания и изменения проекта, а не как надстройка над традиционным CRUD-приложением. Большинство пользовательских действий проходят через [AI Orchestrator](#4-ai-orchestrator) и [Multi-Agent System](#5-multi-agent-system), а не через прямое редактирование данных вручную.

### Modular

Каждая функциональная область (см. [Modules](01-PRODUCT.md#modules)) и каждый агент (см. [AI Agents](01-PRODUCT.md#ai-agents)) реализуются как независимый модуль с чётко определённым контрактом. Модули подключаются и заменяются без переработки платформы в целом (см. [Modular Architecture](01-PRODUCT.md#modular-architecture) и [No Vendor Lock-In](01-PRODUCT.md#no-vendor-lock-in)).

### Event-driven

Изменения в проекте распространяются через события, а не через жёсткие синхронные вызовы между сервисами. Когда структурированные данные проекта меняются, платформа публикует событие, на которое реагируют заинтересованные модули (визуализация, BIM, сметы, документация), обновляя свои производные артефакты (см. [Single Source of Truth](01-PRODUCT.md#single-source-of-truth)).

### API-first

Каждая возможность платформы сначала описывается как API и только затем получает пользовательский интерфейс. Это позволяет одному и тому же функционалу быть одновременно доступным через диалог, рабочее пространство проекта, интеграции и будущие клиентские приложения.

### Vendor-independent

Ни один внешний провайдер — AI-модель, CAD/BIM-система, платёжный сервис — не встроен напрямую в бизнес-логику платформы. Все внешние сервисы подключаются через адаптеры в [Integration Layer](#integration-layer), что позволяет заменять их без изменения ядра платформы (см. [No Vendor Lock-In](01-PRODUCT.md#no-vendor-lock-in)).

### Cloud-native

Платформа проектируется для развёртывания в облаке: сервисы без состояния, горизонтальное масштабирование, управляемые базы данных и хранилища, инфраструктура как код. Локальное состояние не является источником истины ни для одного компонента.

### Multi-tenant

Одна инсталляция платформы обслуживает множество независимых организаций и проектов. Изоляция данных между тенантами обеспечивается на уровне доступа к данным и политик безопасности, а не за счёт отдельных развёртываний под каждого клиента.

### Secure by default

Безопасность не добавляется поверх готовой системы, а закладывается в архитектуру каждого слоя: минимальные права по умолчанию, обязательная аутентификация и авторизация для любого доступа к данным проекта, журналирование действий пользователей и AI-агентов (см. [Security and Privacy by Design](01-PRODUCT.md#security-and-privacy-by-design)).

## High Level Architecture

Запрос пользователя проходит через восемь слоёв платформы — от клиентского приложения до внешних провайдеров:

```
Client Applications
        ↓
    API Gateway
        ↓
  Authentication
        ↓
  AI Orchestrator
        ↓
Multi-Agent System
        ↓
 Business Services
        ↓
      Storage
        ↓
External Providers
```

Каждый слой отвечает за одну зону ответственности и не обходит соседние слои напрямую.

### 1. Client Applications

Пользовательские интерфейсы платформы: веб-приложение, в будущем — мобильные и партнёрские интеграции. Все клиентские приложения обращаются к платформе только через [API Gateway](#2-api-gateway) и реализуют единый интеллектуальный интерфейс (см. [One Intelligent Interface](01-PRODUCT.md#one-intelligent-interface)).

### 2. API Gateway

Единая точка входа для всех запросов: маршрутизация к нужным сервисам, ограничение частоты запросов (rate limiting), версионирование API и агрегация ответов для клиентских приложений.

### 3. Authentication

Проверка личности пользователя или интеграции перед тем, как запрос попадёт дальше в систему. Отвечает за выдачу и проверку сессий/токенов и передачу проверенной личности следующим слоям (подробности — в [Security Layer](#security-layer)).

### 4. AI Orchestrator

Определяет, что нужно сделать с запросом пользователя: интерпретирует намерение, решает, какие агенты и модули должны включиться, и в каком порядке (см. [AI Workflow](01-PRODUCT.md#ai-workflow)). Является техническим воплощением Project Manager Agent на уровне платформы.

### 5. Multi-Agent System

Команда специализированных AI-агентов (см. [AI Agents](01-PRODUCT.md#ai-agents)), каждый из которых решает задачу в своей профессиональной области и обменивается результатами с другими агентами через структурированные данные проекта.

### 6. Business Services

Сервисы, не относящиеся напрямую к AI: управление пользователями и организациями, проектами, файлами, подписками, уведомлениями, правами доступа. Соответствуют переиспользуемому ядру OLNOO (см. [Reusable OLNOO Core](01-PRODUCT.md#reusable-olnoo-core)).

### 7. Storage

Совокупность хранилищ платформы: реляционная база данных, объектное хранилище файлов, векторная база для семантического поиска, кэш и шина событий (подробности — в [Data Layer](#data-layer)).

### 8. External Providers

Внешние AI-модели, CAD/BIM-системы, картографические и погодные сервисы, платёжные провайдеры и другие сторонние сервисы, подключаемые через адаптеры (подробности — в [Integration Layer](#integration-layer)).

## AI Layer

AI Orchestrator и Multi-Agent System опираются на общий набор AI-инфраструктурных компонентов:

### LLM Router

Выбирает конкретную языковую модель для каждой задачи на основе типа запроса, требуемого качества, стоимости и доступности провайдера (см. [Best Tool Wins](01-PRODUCT.md#best-tool-wins)). Позволяет подключать новые модели без изменения логики агентов.

### Prompt Manager

Хранит, версионирует и собирает промпты для агентов и вспомогательных AI-задач. Отделяет формулировки промптов от кода агентов, что позволяет улучшать качество ответов без релиза платформы.

### Memory

Хранит долгосрочный контекст о пользователе, организации и проекте, который должен быть доступен AI между отдельными запросами и сессиями — независимо от истории конкретного диалога.

### Context Builder

Перед каждым обращением к модели собирает релевантный контекст: структурированные данные проекта, историю решений, результаты других агентов и данные из Memory. Ограничивает объём контекста, сохраняя только то, что нужно для конкретной задачи.

### Tool Calling

Единый механизм, через который агенты вызывают внешние инструменты и сервисы — CAD/BIM-системы, расчётные движки, поисковые и картографические API. Инструменты описываются декларативно и одинаково доступны любому агенту, которому они нужны.

### Agent Orchestrator

Технический слой, исполняющий решения AI Orchestrator: запускает нужных агентов, передаёт им контекст, следит за очерёдностью и зависимостями между агентами, обрабатывает ошибки и повторные попытки, собирает результаты обратно в проект.

## Data Layer

### PostgreSQL

Основное реляционное хранилище структурированных данных проекта: сущности, связи, состояния и история изменений (см. [Structured Data First](01-PRODUCT.md#structured-data-first) и [Project Entity](01-PRODUCT.md#project-entity)). Источник истины для всего, что не является бинарным файлом.

### Object Storage

Хранилище файлов и бинарных артефактов: изображения, визуализации, BIM-модели, документы, экспортированные файлы. Каждый объект связан со структурированными данными проекта в PostgreSQL, а не существует изолированно (см. [Single Source of Truth](01-PRODUCT.md#single-source-of-truth)).

### Vector Database

Хранилище векторных представлений для семантического поиска и подбора релевантного контекста — референсов, прошлых решений, документации, нормативных материалов. Используется Context Builder и агентами для поиска релевантной информации, а не для хранения основных данных проекта.

### Cache

Слой кэширования для часто запрашиваемых и дорогих в вычислении данных: результатов AI-запросов, агрегированных представлений проекта, ответов внешних интеграций. Снижает нагрузку на Storage и внешних провайдеров и ускоряет отклик платформы.

### Event Bus

Шина событий, через которую слои и модули платформы уведомляют друг друга об изменениях (см. [Event-driven](#event-driven)). Изменение структурированных данных проекта публикуется как событие, на которое подписываются модули визуализации, BIM, смет и документации.

## Integration Layer

Все внешние сервисы подключаются через адаптеры Integration Layer, что реализует принципы [Vendor-independent](#vendor-independent) и [No Vendor Lock-In](01-PRODUCT.md#no-vendor-lock-in): замена или добавление провайдера не требует изменений в бизнес-логике и агентах.

- **OpenAI** — одна из подключаемых языковых и мультимодальных моделей, доступных через LLM Router.
- **Anthropic** — ещё один провайдер языковых моделей, выбираемый LLM Router по тем же критериям качества и стоимости.
- **Google** — языковые, мультимодальные и картографические возможности, в зависимости от задачи агента.
- **Figma** — импорт и экспорт дизайн-материалов для интерьерных и визуальных решений.
- **GitHub** — хранение и версионирование технических артефактов платформы, интеграция с процессом разработки.
- **BIM software** — экспорт и импорт данных в форматы профессиональных CAD/BIM-систем (например, IFC), используемые BIM Agent.
- **Maps** — данные об участке, местности и окружении, используемые Urban Planning Agent и Landscape Agent.
- **Weather** — климатические данные, используемые Architect Agent, MEP Engineering Agent и Sustainability Agent.
- **Payment providers** — обработка платежей и подписок для Business Services.

## Security Layer

### Authentication

Подтверждение личности пользователя, сервиса или интеграции. Поддерживает независимые от отдельных провайдеров методы входа, чтобы не создавать зависимость от одного механизма аутентификации.

### Authorization

Определяет, какие действия и данные доступны конкретному пользователю в зависимости от его роли и участия в проекте (см. [Progressive Complexity](01-PRODUCT.md#progressive-complexity)). Проверяется на каждом обращении к данным проекта, а не только на уровне интерфейса.

### Audit Log

Журналирует действия пользователей и AI-агентов, влияющие на структурированные данные проекта: кто или какой агент внёс изменение, когда и на основании чего (см. [Explainable Decisions](01-PRODUCT.md#explainable-decisions)).

### Encryption

Данные проекта шифруются при хранении и при передаче между слоями платформы и внешними провайдерами. Особое внимание уделяется персональным данным и коммерчески чувствительной информации проектов.

### Secrets

Ключи доступа к внешним провайдерам, учётные данные баз данных и другие секреты хранятся в выделенном защищённом хранилище секретов, а не в коде или конфигурационных файлах платформы, и не доступны напрямую бизнес-логике или AI-агентам.

## Scalability

Платформа масштабируется горизонтально: каждый слой (API Gateway, AI Orchestrator, Multi-Agent System, Business Services) состоит из сервисов без внутреннего состояния, которые можно запускать в нескольких экземплярах за балансировщиком нагрузки.

Состояние выносится в Storage: PostgreSQL, Object Storage, Vector Database и Cache масштабируются независимо от вычислительных слоёв. Event Bus развязывает модули между собой, позволяя увеличивать пропускную способность одного модуля (например, Visualization Agent под высокой нагрузкой) без изменения остальных.

Такой подход позволяет платформе расти как по числу одновременных пользователей и организаций (см. [Multi-tenant](#multi-tenant)), так и по числу одновременно обрабатываемых проектов и агентов, не меняя архитектуру ядра.

## Future Architecture

Архитектура рассчитана на подключение новых AI-моделей, агентов и сервисов без изменения ядра платформы:

- новые языковые и мультимодальные модели подключаются через LLM Router как дополнительные варианты выбора для существующих задач;
- новые специализированные агенты подключаются к Multi-Agent System как дополнительные участники [AI Workflow](01-PRODUCT.md#ai-workflow), не требуя изменений в уже существующих агентах;
- новые внешние сервисы и CAD/BIM-системы подключаются как дополнительные адаптеры в Integration Layer;
- новые функциональные модули (см. [Modules](01-PRODUCT.md#modules)) подключаются как независимые компоненты Business Services и Multi-Agent System.

Это соответствует принципам [No Vendor Lock-In](01-PRODUCT.md#no-vendor-lock-in) и [Continuous Improvement](01-PRODUCT.md#continuous-improvement): качество и возможности платформы растут со временем, а пользовательский опыт и ядро платформы остаются стабильными.

## Technology Stack (MVP)

Для реализации слоя [Client Applications](#1-client-applications) выбран стек Next.js (App Router) + TypeScript + Tailwind CSS — начиная с версии платформы, где появился первый функциональный прототип (см. [05-FRONTEND.md](05-FRONTEND.md) для деталей по мере их проработки). Первый серверный фундамент использует Supabase Auth через `@supabase/ssr`: cookie-сессии обновляются в Next.js Proxy, а защищённые серверные операции проверяют JWT через `getClaims()`. PostgreSQL, RLS и Object Storage будут подключены только после отдельного утверждения миграции; до этого IndexedDB остаётся временным MVP-хранилищем проектов.
