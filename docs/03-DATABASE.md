## Применённая схема

`supabase/migrations/202607200001_backend_foundation.sql` применена к проекту **architect-olnoo** (ref `iaomlfkcbjeqpgnbpwxs`) и включает:

- workspaces и роли участников (`owner`/`editor`/`viewer`);
- проекты, исходные файлы и подтверждённые ракурсы;
- концепции, связанные исправленные версии и историю версий;
- попытки AI-генерации и события проекта;
- обратную связь по концепциям и ключи идемпотентного импорта локальных записей;
- приватный Storage bucket `project-assets`;
- RLS-политики, ограничивающие данные участниками соответствующего workspace/project.

`supabase/migrations/202607200002_fix_workspace_select_visibility.sql` исправляет найденный при верификации баг: PostgreSQL требует, чтобы строка, вставленная с `RETURNING`, также проходила SELECT-политику таблицы. `workspaces_select` изначально зависела только от `workspace_members`, которую заполняет `AFTER INSERT`-триггер — его результат не гарантированно виден тому же самому implicit SELECT-check в той же команде, поэтому самый первый `insert(...).select()` в `ensureWorkspace()` (`src/lib/server/project-import-repository.ts`) падал с ошибкой RLS для каждого нового пользователя. Политика дополнена условием `owner_user_id = (select auth.uid())`.

Бинарные файлы не хранятся в PostgreSQL: таблица `project_files` содержит только метаданные и путь, а реальные байты размещаются в приватном Supabase Storage по схеме `<workspace_id>/<project_id>/<opaque_file_name>`.

Проект architect-olnoo на момент применения уже содержал несвязанную пустую схему-заготовку (organizations/profiles/tasks/project_comments/...); перед применением миграции она была снята тем же файлом 202607200001 (см. комментарий "LEGACY SCHEMA CLEANUP" в начале миграции) — схема-бэкап снят в `supabase/backups/pre-cleanup-architect-olnoo-20260720.sql`.

## Проверено end-to-end

- Аутентификация: signup/signin через выделенные тестовые аккаунты.
- Анонимный доступ к таблицам и Storage — запрещён (401/403).
- Изоляция между пользователями на уровне workspace/project/Storage — подтверждена (второй пользователь не видит и не может загрузить файлы в чужой workspace).
- Полный цикл синхронизации (prepare → прямая загрузка в Storage → complete) на синтетическом проекте с синтетическим изображением; повтор того же импорта не создаёт дублей ни в одной таблице и не меняет локальную IndexedDB-копию.

Клиент и сервер поддерживают возобновляемый импорт: повтор использует те же `client_import_key` и Storage paths, не удаляет локальные данные и не запускает AI.
