# Security

**Version:** 0.4.0
**Status:** In Progress — authentication and data-isolation controls verified against architect-olnoo
**Last Updated:** 2026-07-20

> Описывает модель угроз, меры безопасности и практики защиты данных пользователей в проекте Architect OLNOO.

---

## Назначение документа

Этот документ фиксирует подход к безопасности: аутентификацию, авторизацию, защиту данных, модель угроз и compliance-требования.

## Аутентификация

- Используется Supabase Auth с cookie-сессией через `@supabase/ssr`.
- Для защиты страниц и данных сервер проверяет подпись JWT через `getClaims()`, а не доверяет непроверенному `getSession()`.
- Платные AI-маршруты имеют собственную проверку сессии и возвращают `401` неавторизованному пользователю.
- Production без Supabase закрывается с `503`; обход без авторизации существует только в локальном `next dev` на loopback-адресе и не действует для LAN-адреса.
- В клиентском окружении разрешены только URL Supabase и publishable key. Service Role key запрещён.

## Авторизация данных

- Миграция включает RLS для всех проектных таблиц и приватный Storage bucket `project-assets`; анонимный доступ отозван и проверен (401/403 на реальном проекте `architect-olnoo`).
- Доступ определяется членством и ролью в workspace, а связи между сущностями ограничены одним проектом; изоляция между пользователями подтверждена end-to-end двумя выделенными тестовыми аккаунтами (таблицы и Storage).
- Браузер загружает файлы прямо в Supabase Storage под пользовательской RLS-сессией; Service Role key не используется и большие бинарные данные не проходят через Next.js.
- Сервер выдаёт только детерминированные пути текущего проекта, повторно проверяет manifest и подтверждает наличие, размер и MIME-тип загруженных объектов до записи метаданных.
- Локальная IndexedDB-копия не удаляется при успехе или ошибке синхронизации — подтверждено на синтетическом проекте, включая повтор без дублей.
