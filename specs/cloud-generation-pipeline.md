# Cloud Generation Pipeline

**Version:** 0.1.0
**Status:** Implemented — automated tests only, no live authenticated verification
**Last Updated:** 2026-07-20

> Платная генерация и исправление концепций для облачных (Supabase) проектов, а также восстанавливаемое состояние платной попытки, переживающее сбой Storage/PostgreSQL после успешного ответа провайдера.

---

## Назначение документа

Расширяет `specs/server-project-repository.md` (чтение/запись облачных проектов) на создание новых концепций. До этой спецификации серверный пайплайн генерации существовал только для локальных (`local-*`, IndexedDB) проектов — `POST /api/concepts/generate` и `POST /api/concepts/correct` принимали исходные изображения напрямую от клиента. Для облачных проектов это неприемлемо: клиент не должен выбирать, какие байты и по какому Storage-пути отправлять в платный запрос — сервер обязан сам загрузить подтверждённые исходники проекта.

## Новые маршруты

| Маршрут | Метод | Назначение |
|---|---|---|
| `/api/projects/:projectId/concepts/generate` | POST | Платная генерация одной концепции для облачного проекта по его подтверждённому Primary View и брифу |
| `/api/projects/:projectId/concepts/:conceptId/correct` | POST | Платное исправление существующей облачной концепции по замечаниям Quality Gate |

Оба маршрута требуют подтверждённую Supabase-сессию (`resolveProjectRouteContext`, отклоняет `local-development` bypass) и принимают JSON, а не FormData: клиент передаёт только `attemptKey` (UUID), `mode` и (для generate) `autoReview` — никаких изображений и текста брифа. Все исходные данные сервер загружает сам:

- `src/lib/server/cloud-generation-source.ts` — находит подтверждённый Primary View и до двух опорных ракурсов проекта (`loadCloudGenerationSource`, генерация) либо повторно скачивает и обрезает точный исходный ракурс, использованный при создании исправляемой концепции, по её сохранённой `sourceProvenance` (`loadProvenanceSourceImage`, исправление) — не текущие `source_views`, чтобы исправление всегда сверялось с тем же геометрическим эталоном, что и исходная концепция.
- Обрезка выполняется на сервере через `sharp` (`src/lib/server/image-crop.ts`) — браузерный `crop-image.ts` (canvas) недоступен в Route Handler.
- Скачивание — `supabase.storage.from('project-assets').download(path)` под RLS-сессией пользователя; путь всегда берётся из строки `project_files`, найденной запросом, уже ограниченным `projectId` — тело запроса не может указать произвольный Storage-путь.
- Промпт, провайдер, модель-реестр и Reviewer не меняются: `src/lib/server/cloud-generation-runner.ts` вызывает те же `resolveModelSpec`/`getProvider`/`getGeometryReviewer`, что и локальный пайплайн, отдельно от `/api/concepts/generate`, чтобы не рисковать регрессией уже проверенного локального маршрута.
- Исправление всегда запускает Reviewer (как и локальный `/api/concepts/correct`); обычная генерация — по флагу `autoReview` из тела запроса.

## Восстанавливаемое состояние платной попытки

`src/lib/server/generation-attempt-repository.ts` реализует состояния поверх уже применённой таблицы `generation_attempts` (`status` — обычный `text` без `CHECK`, миграция не потребовалась):

`pre-dispatch → dispatched → provider-completed → completed`, с `failed` и `persistence-partial` как альтернативными исходами. `dispatched` — терминальное состояние неопределённости: если провайдер не ответил однозначно (сетевая ошибка, таймаут), статус остаётся `dispatched` и не продвигается дальше — это и есть «попытка могла быть оплачена, а могла и нет». Хорошо понятные отказы провайдера (validation, safety-rejection, quota-exhausted, rate-limit) переводятся в `failed` сразу и не считаются неоднозначными.

Идемпотентность обеспечена структурно, не только статусом: `attempt_key` (генерируется клиентом как UUID, проверяется как валидный UUID перед использованием — он становится частью Storage-пути) — это и есть `client_import_key` для `project_files`/`concepts`/`concept_versions`/`activity_events`, поэтому повторная запись с тем же ключом не может создать дубликат ни в одной таблице и не создаёт второй объект Storage (путь детерминирован из `attemptKey`).

После успешного ответа провайдера сервер немедленно:

1. сохраняет статус `provider-completed` вместе со всеми данными, нужными для сохранения концепции (`markAttemptProviderCompleted` — записывает их в `source_provenance` попытки как JSON-конверт, отдельно от исходной provenance, записанной на этапе `pre-dispatch`);
2. загружает байты в приватный Storage по детерминированному пути `<workspace>/<project>/generation-attempts/<attemptKey>.<ext>`;
3. пишет `project_files`/`concepts`(+`concept_versions` для исправления)/`activity_events`, обновляет `projects.state`/`lifecycle_stage`;
4. помечает попытку `completed`.

Если шаг 2 или 3 не удался — попытка помечается `persistence-partial`, а ответ клиенту всё равно содержит `imageBase64`/`mimeType` (как и локальный пайплайн для download-фолбэка). Повтор с тем же `attemptKey` и полем `retryImageBase64`/`retryMimeType` в теле запроса выполняет только шаги 2–4 заново — **не вызывает провайдера повторно**: сервер читает конверт с метаданными концепции обратно из `generation_attempts.source_provenance` (`parseGenerationPersistPayload`), клиенту нужно прислать только байты изображения, которые он уже держит в памяти вкладки с прошлого ответа. Если попытка уже `completed`, повторный запрос с тем же `attemptKey` идемпотентно возвращает уже сохранённую концепцию без каких-либо записей (`findPersistedConceptByAttempt`).

Начать новую платную попытку с тем же `attemptKey`, пока предыдущая находится в `dispatched` (неопределённый исход), запрос отклоняется (`409 ambiguous-attempt`) — интерфейс требует явного чекбокса подтверждения риска повторной оплаты и создаёт новый `attemptKey`, прежде чем повторить запрос (тот же паттерн, что уже использует локальный мастер и `ConceptCorrectionDialog`).

## Что осталось за рамками этой спецификации

- `variantCount` зафиксирован равным 1 для облачной генерации (в отличие от 1–3 у локального пайплайна) — упрощение, сознательно принятое ради предсказуемого состояния одной попытки на один платный запрос; см. ограничения в отчёте задачи.
- Восстановление после обновления страницы работает только пока байты ещё не покинули память вкладки (`persistence-partial`) или уже лежат в Storage (`completed`/идемпотентный повтор); если браузер потерял и вкладку, и байты до того, как Storage-запись удалась, требуется новая явно подтверждённая платная попытка — это прямо сообщается пользователю (`retry-requires-bytes`), а не выдаётся за гарантированное отсутствие оплаты.
- Живая проверка через реальный Supabase-вход не выполнялась по той же причине, что и в `specs/server-project-repository.md` — только точечные тесты на `node:test` без сети и без обращений к Gemini.
