# Multi-View Source Pipeline

**Version:** 0.3.0
**Status:** Draft — Phase 3 implemented
**Last Updated:** 2026-07-20

> Спецификация конвейера обработки исходных фотографий: локальное хранение реальных байт, разбивка коллажей на отдельные ракурсы и передача в генерацию ровно одного подтверждённого Primary View.

---

## Назначение документа

До Phase 1 исходные файлы (`SourceFile`) существовали в проекте только как метаданные (имя, тип, дата) — сами байты изображения нигде не сохранялись, а мастер отправлял `File`-объекты в генерацию напрямую из состояния React. Этот документ описывает конвейер, который: (1) сохраняет реальные байты исходных фотографий в IndexedDB и переживают обновление страницы; (2) распознаёт, когда одна загруженная фотография на самом деле является вертикальным коллажем из нескольких видов дома, и предлагает разбить её на отдельные ракурсы; (3) даёт пользователю подтвердить эти ракурсы до запуска платной генерации.

## Phase 1 — хранение и подтверждение ракурсов

### Хранение исходных изображений

- Байты исходной фотографии сохраняются в IndexedDB в той же структурно-клонируемой форме `{ imageKey, bytes: ArrayBuffer, mimeType }`, что и байты сгенерированных концепций (`src/lib/concept-image-codec.ts`) — хранение `Blob` напрямую уже приводило к `DataCloneError` в этом проекте, см. CHANGELOG.
- Байты живут в отдельном object store (`source-images`, `src/lib/mvp-local-project-store.ts`), ключ — `${sourceFileId}:image`; JSON-метаданные проекта (`sourceFiles`, `sourceViews`) остаются маленькими и без бинарных полей.
- `SourceFile` дополнен необязательными `imageKey`, `mimeType`, `dimensions`; при чтении проекта (`getLocalProject`) байты подтягиваются обратно как `imageBlob` в памяти — это никогда не сериализуется.

### Детектор коллажей

- `src/lib/collage-detector.ts` — чистая, независимая от DOM функция `detectCollageViews(pixels, options?)`. Анализирует построчную (по Y) статистику отдельных строк изображения (среднее и стандартное отклонение оттенков серого), находит внутренние полосы-разделители (сплошные/почти сплошные полосы, контрастирующие с соседним содержимым с обеих сторон) и строит по ним предполагаемые прямоугольники обрезки.
- Всегда возвращает уверенность (`confidence`, 0–1) для найденного разбиения. Если внутренних полос не найдено, разбиение получилось бы слишком тонким, либо итоговая уверенность ниже порога — функция возвращает один ракурс на всё изображение (`isFallback: true`) вместо того, чтобы навязывать сомнительное разбиение.
- Никогда не изменяет переданный ей буфер пикселей и не имеет побочных эффектов — детерминирована, полностью покрыта тестами на `node:test` без браузера (`src/lib/collage-detector.test.ts`).
- Декодирование `File`/`Blob` в пиксели и обрезка по прямоугольнику происходят в браузере через `<canvas>` (`src/lib/crop-image.ts`) — вызывается только из клиентских компонентов.

### Шаг подтверждения ракурсов (мастер)

- Новый шаг мастера «Ракурсы» (`src/components/projects/source-views-step.tsx`), между «Материалы» и «Пожелания» — работает только с уже загруженными в состоянии мастера `File`, ничего не пишет в IndexedDB и не обращается к API генерации.
- Для каждой загруженной фотографии показывает превью каждого предложенного ракурса, позволяет включать/выключать ракурс, назначать роль (`front`/`side`/`rear`/`detail`/`other`) и выбрать ровно один **основной ракурс (Primary View)** среди всех фотографий проекта.
- Явно указывает пользователю, что в генерацию будет передан только основной ракурс — остальные ракурсы сохраняются как материалы проекта, но не отправляются AI-провайдеру.

### Персистентность ракурсов

- При создании черновика проекта (`createDraftProject`) подтверждённые ракурсы преобразуются в записи `SourceView` через чистую функцию `buildSourceRecords` (`src/lib/source-view-builder.ts`), которая：
  - присваивает стабильные `id` файлам и ракурсам;
  - связывает каждый `SourceView.imageKey` с байтами его исходного файла (обрезка происходит по требованию при показе, оригинал никогда не перекодируется и не изменяется);
  - требует ровно один `isPrimary: true` среди подтверждённых ракурсов (если их больше нуля) — иначе выбрасывает ошибку, а не молча сохраняет неоднозначное состояние.
- `SourceView` хранит: `id`, `sourceImageId`, `crop` (прямоугольник), `order`, `role`, `isPrimary`, `imageKey`.

### Раздел «Исходные материалы»

- `src/components/workspace/file-list.tsx` теперь показывает настоящую сохранённую миниатюру оригинала (не заглушку) и, если для файла есть подтверждённые ракурсы, — их превью с бейджами роли и «Основной».
- Работает после обновления страницы: `getLocalProject` каждый раз заново читает байты из IndexedDB.

## Phase 2 — точный Primary View в генерации

- Перед открытием окна платного запуска мастер находит ровно один подтверждённый Primary View и проверяет его связь с исходным растровым файлом, размеры и границы обрезки. Отсутствующий, дублирующийся или выходящий за границы Primary View блокирует запуск до любого сетевого запроса.
- Частичный ракурс вырезается один раз через `cropImageToBlob`; цельный ракурс использует исходный `File` без повторного кодирования. Получившийся `File` одновременно является источником превью в окне подтверждения и единственным значением поля `images` в платном запросе — пользователь видит именно те пиксели, которые будут отправлены провайдеру.
- Если локальный детектор не видит полос-разделителей (например, фотографии в коллаже состыкованы вплотную), пользователь может явно разделить исходник на 2 или 3 равные вертикальные части. Это локальная операция без AI-запроса; границы покрывают все строки изображения без пересечений и потерь, после чего любой из полученных видов можно назначить Primary View.
- Старый выбор до трёх исходных фотографий в мастере для этого сценария удалён: Phase 2 всегда отправляет ровно один Primary View. Остальные загруженные фотографии, PDF и подтверждённые ракурсы остаются материалами проекта.
- Для платного запроса по-прежнему действует клиентская и серверная проверка размера inline-изображения. Количество вариантов результата (1 или 3) не меняет количество исходных изображений в запросе.
- Запись попытки генерации и каждая сохранённая концепция получают `sourceProvenance`: id исходного файла и ракурса, имя файла, роль, crop-прямоугольник и точные параметры отправленного payload (тип, ширина, высота, размер). Это позволяет связать результат с конкретным подтверждённым видом после перезагрузки проекта.
- API Route Handler, модельный реестр и Gemini-адаптер не менялись: Phase 2 изменяет подготовку и происхождение входного изображения на клиенте, сохраняя существующие гарантии платного запроса, повторного сохранения и восстановления.

## Phase 3 — опорные ракурсы как контекст

- Primary View остаётся первым изображением и единственной целью редизайна. До двух следующих подтверждённых ракурсов по порядку передаются как reference context, поэтому общий запрос остаётся в серверном лимите трёх изображений.
- Каждый файл подготавливается тем же точным crop-конвейером Phase 2. В окне платного подтверждения пользователь видит все фактически отправляемые изображения и различает «Основной · цель» и «Опорный · контекст».
- Клиент передаёт отдельный массив `imageContexts` с ролью (`front`/`side`/`rear`/`detail`/`other`) и назначением (`primary`/`reference`). Сервер повторно валидирует совпадение количества записей, допустимые значения и инвариант «первое изображение — единственный primary».
- Gemini получает текстовую метку непосредственно перед каждым изображением. Промпт требует сохранить камеру, кадрирование и перспективу Image 1, использовать остальные виды только для идентичности здания и материалов, не переключаться на их камеру и не создавать коллаж.
- `sourceProvenance` концепции сохраняет Primary View и массив `referenceViews` с точными id, crop и параметрами каждого отправленного payload.

## Явно вне рамок Phase 3

- Генерация отдельного согласованного результата для каждого ракурса одного здания пока не выполняется: результатом остаётся одно изображение с камеры Primary View.
- Автоматическая геометрическая проверка результата по нескольким видам и 3D-реконструкция не реализованы.
- Не добавлен новый AI-провайдер или модель компьютерного зрения — детектор коллажей полностью локален и работает по пиксельной статистике, без обращения к внешним сервисам.
